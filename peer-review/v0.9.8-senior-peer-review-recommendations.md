## Prioritized Recommended Code Fixes (Do Not Apply in This PR)

### P1-1: Remove hardcoded cryptographic keys and move to secret injection

**Issue:** Multiple modules embed static HMAC keys directly in source code, which can lead to predictable signatures and key leakage risk.

**Evidence:**
- `privacy/audit_logger.py` defines `_HMAC_KEY` as a module constant.
- `clinical-analytics/survival_analysis.py` defines `DEFAULT_HMAC_KEY`.
- `clinical-analytics/consortium_reporting.py`, `clinical-analytics/pkpd_engine.py`, `clinical-analytics/risk_stratification.py`, and `clinical-analytics/trial_data_manager.py` define static keys similarly.

**Recommended fix:**
- Replace static literals with secret-provider/env-driven loading (`PAI_HMAC_KEY`, required in prod).
- Fail closed in production mode when key is unset.
- Keep deterministic test defaults only in tests/fixtures.

---

### P1-2: Strengthen audit integrity chain to prevent record reordering/deletion ambiguity

**Issue:** Audit integrity hashes are per-event but not chained to prior event hash, so event deletion/reordering may not be detectable with high confidence.

**Evidence:**
- `AuditEvent._compute_hash()` currently hashes only local event payload.
- `AuditLogger.verify_integrity()` verifies each event independently.

**Recommended fix:**
- Add `previous_hash` to each persisted audit record.
- Compute `event_hash = HMAC(previous_hash + canonical_payload)`.
- Verify full chain in `verify_integrity()` and flag first tamper index.

---

### P1-3: Add explicit PHI-safe logging guardrails

**Issue:** Error paths log raw exception text from data transformation/detection pipelines; depending on downstream exception content, this can leak sensitive values into logs.

**Evidence:**
- `privacy/de-identification/deidentification_pipeline.py` logs raw exception message on transformation failure.
- `privacy/phi-pii-management/phi_detector.py` contains generic exception handlers that log exception details.

**Recommended fix:**
- Route error logs through a sanitizer helper that strips common PHI patterns.
- Use structured logging fields and avoid interpolating raw payload fragments.
- Add unit tests to validate no PHI-like strings are emitted in failure logs.

---

### P2-1: Replace remaining broad exception handlers with typed exception contracts

**Issue:** Remaining `except Exception` blocks reduce diagnosability and can hide programmer errors.

**Evidence:**
- `q1-2026-standards/objective-2-model-registry/model_validator.py`
- `unification/simulation_physics/isaac_mujoco_bridge.py`
- `unification/cross_platform_tools/model_converter.py`
- `unification/cross_platform_tools/policy_exporter.py`
- `privacy/de-identification/deidentification_pipeline.py`

**Recommended fix:**
- Replace broad catches with explicit tuples (`FileNotFoundError`, `ValueError`, `RuntimeError`, framework-specific errors, etc.).
- Include exception type in output messages for triage consistency.

---

### P2-2: Normalize timestamp formatting policy across modules

**Issue:** The repository is mostly timezone-aware, but timestamp serialization format is inconsistent (`isoformat()`, formatted date-only strings, truncated ISO strings), which complicates cross-module analytics and compliance exports.

**Evidence:**
- Mixed timestamp write paths across `regulatory-submissions/*.py`, `privacy/*.py`, and `clinical-analytics/*.py`.

**Recommended fix:**
- Introduce shared utility (`utils/time.py`) with `utc_now_iso()` and `utc_now_date()`.
- Standardize persisted event timestamps to full ISO8601 UTC with offset.
- Reserve display formatting for rendering-only layers.

---

### P2-3: Unify legacy vs canonical privacy module APIs to reduce drift

**Issue:** Legacy and canonical privacy implementations coexist, increasing risk of inconsistent behavior and duplicated maintenance burden.

**Evidence:**
- Legacy modules remain (`privacy/phi_detector.py`, `privacy/deidentification.py`) while canonical implementations live under subdirectories.

**Recommended fix:**
- Add compatibility wrappers with strict deprecation lifecycle (warn -> block new imports -> remove).
- Publish one canonical import path in docs and tests.
- Add a migration test matrix confirming parity until removal.

---

### P2-4: Add property-based/fuzz tests for de-identification and PHI detection edge cases

**Issue:** Current test suite is extensive, but deterministic fixtures may miss malformed/hostile input edge cases (nested structures, unusual unicode, malformed dates/identifiers).

**Evidence:**
- Strong unit test breadth exists, but no explicit fuzz/property-based coverage in privacy critical paths.

**Recommended fix:**
- Add Hypothesis-driven tests for `deidentification_pipeline` and `phi_detector`.
- Assert invariant: transformed outputs contain no recognized PHI patterns from scanner regex baseline.

---

### P3-1: Centralize secure configuration defaults to avoid security drift

**Issue:** Security-sensitive defaults (keys, retention values, operational toggles) are scattered across modules.

**Evidence:**
- Independent constants and defaults appear across privacy and analytics modules.

**Recommended fix:**
- Create centralized config module with typed settings and environment loading.
- Mark security-sensitive fields as required in production profile.

---

### P3-2: Improve operational observability with structured error codes

**Issue:** Many modules provide human-readable errors but lack stable machine-oriented error codes for monitoring/alert routing.

**Evidence:**
- Error reporting in conversion, validation, and bridge modules is mostly free-text.

**Recommended fix:**
- Add enum-based error codes (e.g., `MODEL_LOAD_FAILED`, `SIM_BACKEND_INIT_FAILED`) to reports/logs.
- Include error code coverage in tests.

---

### P3-3: Add CI gate for “hardcoded secret-like literal” detection

**Issue:** Existing `security_scan.py` catches several patterns but does not fail on likely embedded cryptographic secrets.

**Evidence:**
- Static HMAC key literals are currently present while security scan passes.

**Recommended fix:**
- Extend `scripts/security_scan.py` with a rule to detect key-like constants (`*_KEY = b"..."`, `DEFAULT_HMAC_KEY`, etc.) outside tests/examples.
- Allow inline suppressions with justification comments for approved exceptions.
